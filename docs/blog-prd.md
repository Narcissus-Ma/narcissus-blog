## 2. 项目背景

当前博客基于 Hexo，内容展示成熟，但存在以下限制：

- 内容管理能力受限（草稿、版本、回收站、批量管理能力不足）
- 动态能力有限（后端可扩展性、接口复用、运营配置）
- 前后端不统一（类型共享与接口治理成本偏高）

目标是构建一套可持续迭代的前后端分离博客系统：

- 前端：Cloudflare Pages 部署，保持旧站视觉一致性
- 后端：NestJS + Fastify + Prisma + SQLite，运行在 2G VPS
- 文件：对象存储（R2/OSS）
- 研发方式：pnpm Monorepo + TypeScript 全链路

## 3. 产品目标与成功指标

### 3.1 核心目标

1. 复刻旧站主要视觉体验与浏览路径，降低迁移认知成本。
2. 提供完整后台内容管理能力，支持长期更新运营。
3. 在 2G VPS 成本约束下，保证稳定与性能。

### 3.2 成功指标（上线后 30 天）

1. 可用性：站点可用性 >= 99.9%。
2. 性能：

- 首页 LCP（4G 网络，移动端）<= 2.8s。
- 文章详情首屏可交互 <= 2.5s。

3. 运营效率：

- 新文章从创建到发布全流程 <= 5 分钟。
- 图片上传成功率 >= 99.5%。

4. 质量：

- 严重线上故障（P1）= 0。

## 4. 用户角色

### 4.1 游客（未登录）

- 浏览首页推荐、分类、标签、归档
- 阅读文章、检索内容
- 切换明暗主题

### 4.2 站长/编辑（登录后台）

- 管理文章、分类、标签、封面与媒体资源
- 管理站点配置（导航、推荐位、SEO 字段）
- 查看基础统计与发布状态

## 5. 范围定义

### 5.1 MVP（必须）

1. 前台：首页、文章详情、分类页、标签页、归档页、搜索页。
2. 后台：登录鉴权、文章 CRUD、分类/标签管理、资源上传、站点配置。
3. API：统一鉴权、错误处理、分页与筛选。
4. 部署：前端 Cloudflare Pages、后端 Docker + Nginx + HTTPS。

### 5.2 增强（应做）

1. 草稿/定时发布。
2. 文章推荐位配置（首页 Top 区域）。
3. SEO 扩展（sitemap、RSS、OpenGraph）。

### 5.3 远期（可选）

1. 评论系统接入（Twikoo/Giscus）。
2. 订阅/通知（邮件或 RSS 增强）。
3. 轻量分析面板（PV/UV/热门文章）。

## 6. 功能需求明细

### 6.1 前台（Web）

#### FR-FE-01 首页

- 顶部固定导航：站点名、归档/分类/标签、搜索入口、随机文章、主题切换。
- Hero 区：随机引导卡 + 分类快捷入口（保留旧站风格）。
- 推荐区：置顶/推荐文章卡片。
- 列表区：文章卡片（封面、标题、摘要、发布时间、标签、分类）。
- 支持分页或“加载更多”。

验收标准：

- 与旧站结构一致（导航 + Hero + 分类条 + 卡片列表 + 侧栏）。
- 首屏在移动端、桌面端均可读且无布局错位。

#### FR-FE-02 文章详情

- Markdown 渲染（GFM、代码高亮、图片、表格、引用）。
- 文章信息：标题、发布时间、更新时间、标签、分类。
- 目录（可选固定）与返回顶部。
- 上一篇/下一篇导航。

#### FR-FE-03 分类/标签/归档

- 分类页：按分类聚合文章。
- 标签页：Tag Cloud + 文章列表。
- 归档页：按年月折叠。

#### FR-FE-04 搜索

- 支持关键词搜索标题/摘要/正文。
- 高亮命中片段。
- 无结果提示与建议操作。

#### FR-FE-05 主题与阅读体验

- 明/暗主题切换，支持系统偏好自动跟随。
- 字体、色彩、行高保持旧站阅读习惯。

### 6.2 后台（Admin）

#### FR-AD-01 登录鉴权

- 账号密码登录，JWT 鉴权。
- Access + Refresh Token。
- 登录失败保护（频率限制）。

#### FR-AD-02 文章管理

- 新建/编辑/预览/发布/下线/删除。
- 草稿状态与发布时间。
- Markdown 编辑器（Monaco）+ 封面选择。

#### FR-AD-03 分类与标签管理

- 分类增删改查。
- 标签增删改查。
- 文章与分类/标签关联管理。

#### FR-AD-04 媒体资源管理

- 图片上传对象存储（R2/OSS）。
- 文件大小与类型校验。
- 返回可用 URL（支持 CDN 分发）。

#### FR-AD-05 站点配置

- 导航菜单配置。
- 首页推荐位配置。
- SEO 默认字段（title/description/og:image）。

### 6.3 接口与系统能力

#### FR-SYS-01 API 规范

- RESTful 风格，统一响应结构。
- 支持分页、排序、筛选。

#### FR-SYS-02 错误与认证

- Axios 拦截器统一处理错误码与 token 刷新。
- 后端统一异常过滤器输出。

#### FR-SYS-03 数据一致性

- 前后端共享 DTO/Interface，字段变更可编译期感知。

## 7. 非功能需求

### 7.1 性能

- 前台页面需启用静态资源压缩（gzip/br）。
- 图片优先 WebP/AVIF。
- 列表接口 P95 响应时间 <= 300ms（VPS 本地环境）。

### 7.2 安全

- 禁止不安全函数（如 `eval()`、`with()`）。
- 输入校验 + XSS 基础防护。
- 鉴权接口限流。

### 7.3 可维护性

- TypeScript 优先 `interface` 描述数据结构。
- 状态管理按功能模块拆分，避免单一全局巨型 Store。
- API 按业务域分层组织。

### 7.4 兼容性

- 主流现代浏览器兼容。
- 响应式覆盖：手机（<=768）、平板（769-1200）、桌面（>1200）。

## 8. 数据实体（高层）

1. User：管理员账号信息。
2. Article：标题、摘要、正文、状态、SEO 字段、发布时间。
3. Category：分类信息。
4. Tag：标签信息。
5. MediaAsset：对象存储 URL、类型、大小。
6. SiteSetting：导航、推荐位、主题参数。

## 9. 约束与依赖

1. 成本约束：2G VPS，后端服务需控制内存峰值。
2. 部署约束：前端 Cloudflare Pages（需 `_redirects` 处理 SPA 路由）。
3. 存储约束：媒体文件不落 VPS 本地，统一走对象存储。
4. CDN 策略：静态第三方资源优先国内可用 CDN（BootCDN、又拍云等），并保留降级兜底。

## 10. 验收标准（上线门槛）

1. 功能验收：MVP 清单全部通过联调。
2. 设计验收：首页视觉结构与旧站一致度 >= 90%（布局、色彩、组件交互）。
3. 质量验收：ESLint + Prettier + 单元测试 + 核心链路冒烟测试通过。
4. 运维验收：部署脚本可重复执行，支持回滚到上一版本。

## 11. 里程碑（建议）

1. M1（第 1-2 周）：基础工程 + 登录 + 文章 CRUD API。
2. M2（第 3-4 周）：前台首页/详情/分类标签归档完成。
3. M3（第 5-6 周）：后台管理、媒体上传、站点配置完成。
4. M4（第 7 周）：部署联调、性能优化、上线验收。

## 12. 风险与应对

1. 旧站 UI 复刻偏差：先冻结 UI Token + 页面骨架后再开发。
2. 2G VPS 内存波动：Fastify + SQLite + 资源限制 + 日志控量。
3. 对象存储兼容问题：先做 R2，保留 OSS 适配层。
4. SEO 风险：增加预渲染或静态快照策略。

## 13. 不在本期范围

1. 多角色权限系统（仅单管理员）。
2. 社区互动体系（点赞、收藏、私信）。
3. 多语言国际化。
